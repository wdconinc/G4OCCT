# G4OCCT — NIST CTC Geometry Fixtures

G4OCCT includes 11 STEP geometry fixtures drawn from the **NIST MBE PMI
Conformance Test Case (CTC)** suite — a publicly available collection of
AP203-format STEP files released by the National Institute of Standards and
Technology (NIST) to benchmark CAD interoperability and model-based enterprise
(MBE) workflows.  These fixtures exercise G4OCCTSolid with real-world product
geometry that has no direct Geant4 analytic equivalent, making them an
important stress test for the STEP import pipeline.

NIST MBE PMI project: <https://www.nist.gov/programs-projects/mbe-pmi-validation-and-testing>

---

## What is the CTC?

The NIST MBE PMI Conformance Test Case suite consists of AP203 geometry-only
STEP files that represent simplified but realistic mechanical-part geometries.
The suite was designed to expose differences in how CAD systems import, export,
and interpret STEP data, particularly for assemblies with compound B-rep solids.

For G4OCCT the CTC parts serve a different purpose: they exercise
`G4OCCTSolid`'s ability to import multi-shell compound assemblies, compute
bounding volumes, and respond correctly to Geant4's `Inside()` query — all
without any reference native Geant4 solid to compare against.

Key properties of the CTC fixtures used in G4OCCT:

- **Format:** AP203 geometry-only STEP (no PMI annotations, no colour, no
  material attributes)
- **Topology:** compound assemblies (`MANIFOLD_SOLID_BREP` collections) with
  multiple shells
- **Origin:** downloaded from the NIST SFA GitHub repository at CI time
- **Naming convention:** `nist-ctc-NN` where `NN` runs from `01` to `11`

## Current Validation Status

Unlike every other fixture family in G4OCCT, the NIST CTC parts have **no
native Geant4 solid equivalent**.  The usual validation strategy — comparing
`G4OCCTSolid` results against a reference `G4VSolid` — cannot be applied here.

### What IS tested

| Check | Method | Status |
|---|---|---|
| STEP import | `G4OCCTSolid` constructor does not throw | ✅ Enabled |
| `Inside()` volume estimate | Monte-Carlo point sampling against bounding box | ✅ Enabled via `test_nist_ctc_inside_volume` |

### What is NOT tested (and why)

| Check | Reason |
|---|---|
| Ray-comparison against native Geant4 | No native Geant4 equivalent exists |
| Geometry validation (strict) | Too slow for compound assemblies; disabled in CI |
| Timing benchmarks | Run in CI via `bench_navigator`, but no pass/fail thresholds are enforced |
| Point cloud visualisation | Exported in CI as `reports/point_cloud_viewer.html` for manual inspection only |
| Rendered images | Generated in CI as `reports/geometry-images.html` for manual inspection only |

Because there is no native Geant4 reference solid, `test_nist_ctc_inside_volume`
instead uses the imported OCCT shape as the reference: it computes a volume
using `BRepGProp::VolumeProperties()` and then estimates the Geant4 volume via
Monte-Carlo point sampling against the bounding box and `Inside()`. The test
requires the Monte-Carlo estimate to agree with the OCCT reference volume
within a specified tolerance, which both validates `Inside()` for these parts
and checks that the imported geometry and bounding box are consistent.

## Fetching the Fixtures

STEP files are **not** stored in the G4OCCT repository.  CI downloads them at
build time from the NIST SFA GitHub repository using the `fetch-nist-ctc`
composite action.

For local development, run the fetch script from the repository root:

```bash
bash src/tests/fixtures/geometry/nist-ctc/fetch.sh
```

This places the STEP files under
`src/tests/fixtures/geometry/nist-ctc/` where the CMake test machinery
expects them.
